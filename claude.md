# IAM Policy Sentinel - Project Context

## Project Name
IAM Policy Sentinel

## Mission
Build a fully offline AWS IAM policy validation and least-privilege enforcement tool that operates without live AWS calls, using local SQLite databases for policy_sentry data and resource inventory.

## Technology Stack
- Language: Python 3.9+
- Database: SQLite3
- Testing: pytest
- Type Checking: mypy
- Code Style: dataclasses, type hints, Google-style docstrings
- No AWS SDK (boto3) - fully offline operation

## Architecture Overview

### Four-Step Pipeline
1. VALIDATE: Parse policy JSON, verify actions against local DB, three-tier classification
2. ANALYZE: Map intent to access levels, detect risks, identify missing permissions
3. REWRITE: Generate least-privilege policy with specific ARNs or placeholders
4. SELF-CHECK: Re-validate rewritten policy, verify functional completeness

### Agent Architecture (3-Agent System)
- Agent 1: Web Scraper - Fetches policy_sentry and AWS Service Authorization data
- Agent 2: Code Writer - Implements all 4 pipeline components + database + CLI
- Agent 3: Validator - Validates code against 7 test case categories

### Database Schema Strategy

**IAM Actions Database:**
```sql
CREATE TABLE services (
    service_prefix TEXT PRIMARY KEY,
    service_name TEXT NOT NULL
);

CREATE TABLE actions (
    action_id INTEGER PRIMARY KEY,
    service_prefix TEXT,
    action_name TEXT,
    access_level TEXT, -- List, Read, Write, PermissionsManagement, Tagging
    resource_types TEXT, -- JSON array
    condition_keys TEXT, -- JSON array
    UNIQUE(service_prefix, action_name)
);

CREATE TABLE resource_types (
    resource_type_id INTEGER PRIMARY KEY,
    service_prefix TEXT,
    resource_type TEXT,
    arn_pattern TEXT
);

CREATE TABLE metadata (
    key TEXT PRIMARY KEY,
    value TEXT
);
```

**Resource Inventory Database:**
```sql
CREATE TABLE resources (
    resource_id INTEGER PRIMARY KEY,
    service_prefix TEXT,
    resource_type TEXT,
    resource_arn TEXT UNIQUE,
    resource_name TEXT,
    region TEXT,
    account_id TEXT,
    metadata TEXT -- JSON
);

CREATE INDEX idx_service ON resources(service_prefix);
CREATE INDEX idx_type ON resources(resource_type);
```

## Key Design Decisions

### Decision 1: SQLite Over External APIs
**Rationale:** Fully offline operation, no network dependencies, fast local queries, portable single-file databases.

### Decision 2: Three-Tier Action Classification
**Rationale:** Safely handle new AWS actions without breaking validation, human confirmation for uncertainty.

### Decision 3: Intent-to-Access-Level Mapping
**Rationale:** Core translation mechanism from natural language ("read-only") to precise IAM permissions (List + Read).

### Decision 4: Placeholder ARN Strategy
**Rationale:** When resource inventory unavailable, use clearly marked placeholders rather than wildcards to maintain least-privilege principle.

### Decision 5: Separate Read/Write Statements
**Rationale:** Clearer policy structure, easier debugging, better audit trail.

### Key Learning: Exception Handling Pattern

The critical bug taught an important lesson about Python exception handling:

**Problem Pattern:**
```python
try:
    conn = sqlite3.connect(path)  # If this fails, conn never exists
finally:
    if conn:  # UnboundLocalError if connection failed
        conn.close()
```

**Correct Pattern:**
```python
conn = None  # Initialize before try
try:
    conn = sqlite3.connect(path)
finally:
    if conn:  # Safe - conn is always defined
        conn.close()
```

This pattern is now applied consistently across all database modules.

### Key Learning: Dictionary Duplicate Keys

Python silently overwrites duplicate dictionary keys - no error, no warning.
This caused a subtle data loss bug where companion permissions were silently dropped.

**Problem Pattern:**
```python
RULES = {
    'lambda:CreateFunction': CompanionPermission(companions=['logs:*']),
    # ... many lines later ...
    'lambda:CreateFunction': CompanionPermission(companions=['ec2:*']),  # Overwrites!
}
```

**Fix:** Merge into single entry or use a validation check for duplicate keys.
